VEREDICTO GLOBAL
----------------
Postura de seguridad en RED     : INSUFICIENTE (auth, rate limit, TLS ausentes)
Postura en modo desktop local   : ACEPTABLE CON RESERVAS (confianza en el host)
Postura vs auditoría anterior   : MEJORADA en contención P0; gaps estructurales persisten

NeuroMonitor expone telemetría detallada del sistema operativo: hostname, puntos
de montaje, dispositivos de bloque, modelo GPU, núcleos CPU y patrones de carga
en tiempo real. No existe autenticación ni autorización en la API cuando se
habilita. El producto asume un entorno de confianza (máquina del usuario).

CONTEXTO ARQUITECTÓNICO
-----------------------
Flujo desktop (producción actual):
  MonitorService → NeuroMonitorApplication (poll loop)
                → MetricsHub (pub/sub in-process)
                → HistoryBuffer + evaluate_js → frontend React
                → NeuroMonitorBridge (pywebview.api RPC)

Flujo API (opcional, opt-in):
  main.py --api → run_api_server() → uvicorn → FastAPI create_app()
  Endpoints: GET /health, GET /metrics/snapshot
  WebSocket: DESHABILITADO en paquete neuromonitor/; ACTIVO en src/neuromonitor/

Superficie de ataque desktop:
  - file:// assets/index.html en pywebview
  - pywebview.api.* invocable desde contexto JS
  - window.onNewSnapshot global (evaluate_js push)
  - Sin puerto TCP en modo desktop (riesgo de red BAJO directo)

Superficie de ataque API (si habilitada):
  - HTTP sin TLS propio; sin token; sin rate limit
  - /docs, /redoc, /openapi.json expuestos (FastAPI default)
  - Bind 127.0.0.1 por defecto (mitiga exposición LAN, no localhost abuse)
